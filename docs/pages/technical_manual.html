📘 Manual Técnico

Arquitectura de la API y conexión de servicios del Frontend con el backend FastAPI.

1. Arquitectura API (Backend)

1.1 Estructura de Endpoints

El router maestro en backend/app/api/router.py organiza las rutas bajo los siguientes recursos:

  • /stores/: Gestión de tiendas y locales.
  • /categories/: Clasificación de productos.
  • /products/: Plantillas de productos y SKUs.
  • /employees/: Gestión de personal y accesos.
  • /roles/: Permisos del sistema.

1.2 Validación y Tipado

Usamos **SQLModel** para garantizar que los datos que entran (Create) y salen (Response) cumplen estrictamente con los esquemas definidos en backend/app/models/. Esto genera automáticamente la documentación OpenAPI en /docs.

1.3 Capa CRUD

La lógica de persistencia está aislada en backend/app/crud/. Cada función recibe una Session de base de datos, lo que facilita el testing unitario sin necesidad de levantar el servidor web.

2. Conexión de servicios Frontend

2.1 Cliente API Centralizado

Localizado en frontend/src/services/api.ts. Centraliza el baseURL y los interceptores para manejar errores globales o inyectar cabeceras de autorización.

2.2 Servicios por Recurso

Cada entidad tiene su propio servicio en frontend/src/services/ (ej: stores.service.ts), exportando funciones asíncronas que devuelven Promesas tipadas con interfaces de TypeScript.

2.3 Implementación en UI

Las páginas administrativas en frontend/src/pages/admin/ utilizan estos servicios dentro de useEffect o callbacks de eventos para sincronizar el estado de la UI con el servidor.

3. Estándares Técnicos

  • Formato: JSON para todas las peticiones y respuestas.
  • Codificación: UTF-8.
  • Errores: Uso de códigos de estado HTTP semánticos (404 Not Found, 409 Conflict, 422 Validation Error).
  • Audio: El frontend utiliza la ruta /sounds/notification-chime.mp3 para feedback sonoro en operaciones exitosas.
