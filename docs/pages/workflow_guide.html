🔄 Flujo de Trabajo (Jira y Git)

Reglas obligatorias de equipo sobre cómo gestionar tareas, ramas y despliegues.

1. Gestión de Tareas (Jira)

Todo el desarrollo en Stocker está gobernado por Jira. No se debe escribir código fuente sin que exista una tarea (issue) o subtarea asignada en el tablero activo.

🔹 Integración Jira - Git

  1. Al asignarte una tarea o subtarea en Jira, usa la integración nativa para hacer clic en "Crear Rama" (Create Branch) directamente desde el panel de Jira.
  2. En la ventana emergente, selecciona el repositorio (Stocker) y la rama base (main). Importante: Como Jira genera el nombre automáticamente (ej. P3-165-FRONT...), debes teclear manualmente el prefijo feature/, bugfix/ o hotfix/ justo delante del texto generado antes de crear la rama.
  3. Desarrolla en local dentro de esa rama.
  4. Confirmación (Commit): Todo commit DEBE contener la clave de Jira al inicio y seguir la convención del proyecto.

    📝 Ejemplo:
    git commit -m "P3-165 [FRONT] Inicializar proyecto React con Vite y Tailwind"
  5. Desde el mismo ticket de Jira (o GitHub/GitLab), genera el Pull Request (PR) cuando hayas terminado.

2. Protección de la Rama Principal (main)

Para asegurar que Stocker nunca se rompe ("Merge Hell" o código no testeable), la rama main está bloqueada.

❌ Prohibido (Direct Push):
Nunca debes ejecutar git push origin main ni programar directamente sobre esa rama.

Pasos para Administradores: Cómo configurar esta regla en GitHub

El administrador del repositorio debe seguir estos pasos exactos una sola vez para blindar el proyecto:

  1. Ve a la pestaña Settings del repositorio en GitHub.
  2. En el menú lateral izquierdo, haz clic en Branches.
  3. Dentro de "Branch protection rules", pulsa el botón Add branch protection rule.
  4. Para el patrón del nombre de la rama (Branch name pattern), escribe: main
  5. Marca la casilla Require a pull request before merging.
  6. Asegúrate de marcar Require approvals (mínimo 1).
  7. Es CRÍTICO marcar la casilla secundaria: Require review from Code Owners. Esto activará nuestro archivo .github/CODEOWNERS.
  8. Haz clic en el botón verde inferior Create.

3. CODEOWNERS (Propietarios de Código)

Hemos implementado el archivo .github/CODEOWNERS en la raíz del repositorio. Este archivo actúa como un algoritmo de automatización silencioso:

*   @vinanrra @aiuoki @franarxs @wejhvabewjty

¿Qué provoca esto?

  • Cuando cualquier miembro del equipo abra un Pull Request desde Jira, las plataformas detectarán automáticamente que estas 4 personas son propietarias de código.
  • Se les añadirá al instante a la lista de "Reviewers" sin que tú tengas que agregarlos a mano.
  • Cualquiera de los 4 puede aprobar (Approve) el Pull Request, permitiendo al autor fusionar el botón "Merge to main".
