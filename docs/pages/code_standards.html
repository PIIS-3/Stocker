📖 Guía de Estándares de Código

Define las reglas de formato y las herramientas de calidad del proyecto. Aplica automáticamente en cada Pull Request a través de los GitHub Actions.

Preparación del Entorno (VS Code / Cursor): Instala las extensiones EditorConfig, Prettier y ESLint. Activa "Format on Save" para que el editor aplique las reglas automáticamente al guardar.

1. Herramientas de Estandarización

⚙️ EditorConfig — .editorconfig (Repositorio completo)

Establece las reglas más básicas de formato. Las aplica tu editor automáticamente al escribir, sin necesidad de ejecutar ningún comando.

OpciónValorExplicación
charsetutf-8Codificación estándar que soporta todos los idiomas y símbolos.
end_of_linelfFinales de línea Unix. Evita problemas con Windows (CRLF) que rompen scripts y crean falsos cambios en Git.
indent_stylespaceUsa espacios en lugar de tabulaciones para consistencia entre editores.
indent_size2 (JS/TS), 4 (Python)2 espacios para frontend; 4 para Python según el estándar PEP 8.
trim_trailing_whitespacetrueElimina los espacios invisibles al final de cada línea que generan ruido en los diffs de Git.
insert_final_newlinetrueAñade una línea vacía al final de cada archivo. Requerido por el estándar POSIX y por EditorConfig Checker.

Workflow que lo verifica: lint-global.yml → ejecuta editorconfig-checker en cada PR.

✨ Prettier — .prettierrc + .prettierignore (Solo Frontend)

Reformatea el código visualmente de forma automática. Se ocupa del aspecto del código (comillas, ancho de línea, comas finales), no de su lógica.

OpciónValorExplicación
semitruePunto y coma obligatorio al final de cada sentencia.
singleQuotetrueComillas simples en JS/TS: 'texto' en lugar de "texto".
tabWidth2Indentación de 2 espacios. Coherente con .editorconfig.
trailingCommaes5Añade coma final en objetos y arrays multilínea (reduce el ruido en diffs de Git al añadir nuevas líneas).
printWidth100Máximo 100 caracteres por línea antes de hacer salto automático.
jsxSingleQuotefalseEn JSX/HTML, usa comillas dobles: className="btn" (convención HTML estándar).
arrowParensalwaysParéntesis siempre en arrow functions: (x) => x en vez de x => x.
endOfLinelfFinales de línea Unix. Coherente con .editorconfig.

Archivos ignorados por .prettierignore: dist/, node_modules/, *.min.*, *.map.

Workflow que lo verifica: lint-frontend.yml → ejecuta npm run format:check.

🚨 ESLint — eslint.config.js (Solo Frontend)

Analiza la lógica del código y detecta errores, variables sin usar y malas prácticas de React/TypeScript. Complementa a Prettier: uno revisa el aspecto, el otro la lógica.

ReglaNivelExplicación
no-console warn Avisa si dejas un console.log olvidado. Solo se permiten console.warn y console.error. Para desactivar puntualmente: // eslint-disable-next-line no-console.
no-unused-vars warn Avisa si declaras una variable y no la usas. Si es intencional, usa el prefijo _: const [_state, setState] = useState().
consistent-type-imports warn Obliga a usar import type para interfaces y tipos de TypeScript. Permite a Vite eliminarlos completamente al compilar, reduciendo el bundle final.
no-cascading-renders error Evita llamar a setState de forma síncrona dentro de un useEffect. Inicializa el estado directamente en useState(() => ...) para mejorar el rendimiento.

Workflow que lo verifica: lint-frontend.yml → ejecuta npm run lint.

2. Comandos de package.json — Cómo Ejecutarlos Manualmente

Todos los comandos se ejecutan desde la carpeta frontend/.

Comando Script interno Cuándo usarlo
npm run dev vite --host Arrancar el servidor de desarrollo local. El --host lo expone en la red local (necesario para Docker). Accede en http://localhost:5173.
npm run build tsc -b && vite build Compilar para producción. Primero comprueba los tipos con TypeScript (tsc -b) y luego genera los archivos optimizados en dist/. Solo necesario para desplegar.
npm run lint eslint . Revisar errores de lógica y malas prácticas en todo el código TypeScript. Para corregir automáticamente lo posible: npm run lint -- --fix.
npm run format:check prettier --check . Verificar que el formato del código cumple las reglas (sin modificar archivos). Para aplicar el formato automáticamente: npx prettier --write ..
npm run preview vite preview Previsualizar localmente el bundle de producción ya compilado. Requiere haber ejecutado npm run build antes.
Flujo recomendado antes de hacer un commit: 
npx prettier --write .   # 1. Aplica formato automáticamente
npm run lint             # 2. Revisa errores de lógica
npm run format:check     # 3. Confirma que el formato está correcto

3. GitHub Actions — Verificación Automática en PRs

Cada Pull Request ejecuta automáticamente los siguientes workflows. Si alguno falla, el merge queda bloqueado.

🤖 lint-frontend.yml

Se activa en: Push a main y en cualquier Pull Request.

  1. Descarga el código del repositorio.
  2. Instala Node.js 20 con caché de dependencias: si package-lock.json no ha cambiado, GitHub reutiliza node_modules de la ejecución anterior (pasa de ~2 min a ~15 segundos).
  3. Instala dependencias con npm ci (reproducible, sin actualizar versiones).
  4. Ejecuta npm run lint (ESLint).
  5. Ejecuta npm run format:check (Prettier).
🤖 lint-global.yml

Se activa en: Push a main y en cualquier Pull Request.

  1. Descarga el código del repositorio.
  2. Ejecuta editorconfig-checker: verifica que todos los archivos del repositorio cumplen con las reglas de .editorconfig (indentación, finales de línea, línea final en blanco).

4. Nombrado de Variables (Naming Conventions)

🐍 Backend (Python)
  • Variables y funciones: snake_case (ej: user_name, get_product())
  • Clases (Modelos): PascalCase (ej: ProductTemplate, EmployeeBase)
  • Constantes: UPPER_SNAKE_CASE (ej: MAX_RETRIES = 3)
⚛️ Frontend (React/TypeScript)
  • Variables y funciones: camelCase (ej: userName, handleSubmit())
  • Componentes React: PascalCase (ej: ProductsList(), AdminLayout())
  • Interfaces de TypeScript: PascalCase (ej: ApiProduct)

5. Reglas de Documentación y Comentarios

💬 Docstrings vs Comentarios (#)

En Python, hay dos tipos de anotaciones con propósitos distintos:

Docstring """..."""

Documentación pública de la función. Aparece en Swagger y en el help() de Python. Solo debe describir qué hace la función, qué recibe y qué devuelve.

Comentario # ...

Nota interna para quien lee el código fuente. Explica el por qué de una decisión técnica, no el qué. Va dentro del cuerpo de la función.

🌍 Idioma de los Comentarios

El código (nombres de variables, funciones, clases) siempre en inglés. Los comentarios y docstrings siempre en español.

# Disable user input if loading
if (isLoading) { ... }
# Deshabilita el input mientras la API responde
if (isLoading) { ... }
