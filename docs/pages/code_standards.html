📖 Guía de Estándares de Código

Esta guía define las reglas de formato estático y las convenciones establecidas para evitar conflictos de equipo.

Preparación del Entorno (VS Code): Es obligatorio instalar las extensiones EditorConfig, Prettier y ESLint, y tener activado "Format on Save" en los ajustes.

1. Herramientas de Estandarización

⚙️ EditorConfig (Aplica a todo el repositorio)

Se encarga de homogeneizar el formato más básico a nivel de editor, sin importar si usas VS Code, WebStorm o PyCharm.

  • Finales de línea: LF (formato Unix). Bloquea que un desarrollador de Windows introduzca finales de línea CRLF que rompen scripts de bash y causan falsos difs en Git.
  • Indentación: 4 espacios estrictos para Python (PEP 8), 2 espacios para Frontend y config files.
  • Automático: Recorta espacios vacíos al final de las líneas de código y añade una nueva línea en blanco al final de cada archivo.
✨ Prettier (Solo Frontend)

Se enfoca ciegamente en el envoltorio visual del código: longitud de líneas, comillas, etc.

  • Comillas simples: Forzadas en JS/TS en lugar de dobles (excepto en JSX/HTML).
  • Punto y coma (;): Obligatorios en sentencias.
  • Ancho máximo: El código no debe superar los 100 caracteres antes de saltar la línea automáticamente.
🚨 ESLint (Solo Frontend)

Revisa la _lógica_ y avisa de malas prácticas o errores de TypeScript.

  • Eliminación de `console.log`: Lanza advertencias (amarillo) si olvidas un console.log.
    Excepción: Si es necesario (como el logger global de la API), usar `// eslint-disable-next-line no-console`.
  • Imports de Tipos: Obliga a usar import type cuando solo importas interfaces TypeScript, para que Vite las purgue mejor al construir.
  • Variables no usadas: Lanza un aviso si declaras algo y nunca lo usas. Si es intencionado, predícelo con guión bajo (ej: _req o _res).

2. Reglas de Documentación

Una base de código mantenible se autodescribe, pero requiere claridad humana para las decisiones inusuales.

💬 Idioma de los Comentarios

A diferencia del código (que va siempre cifrado en inglés), el 100% de los comentarios de equipo deben ir en Español.

// Disable user input if loading occurs
if (isLoading) { ... }
// Deshabilita el input del usuario mientras la llamada a la API está en curso
if (isLoading) { ... }
🤔 Comentarios Descriptivos (El Porqué)

El código explica qué pasa, pero el comentario debe explicar el por qué (motivos de negocio, riesgos técnicos, arquitecturas).

# Configura el echo a True
engine = create_engine(db_url, echo=True)
# echo=True le indica a SQLAlchemy que imprima
# en consola las consultas SQL puras.
# Activarlo solo para depuración en desarrollo local.
engine = create_engine(db_url, echo=True)

3. Nombrado de Variables (Naming Conventions)

Seguir el estándar histórico de los respectivos lenguajes.

🐍 Backend (Python)
  • Variables y funciones: snake_case (ej: user_name, get_product())
  • Clases (Modelos): PascalCase (ej: ProductTemplate, EmployeeBase)
  • Constantes: UPPER_SNAKE_CASE (ej: MAX_RETRIES = 3)
⚛️ Frontend (React/TypeScript)
  • Variables y funciones: camelCase (ej: userName, handleSubmit())
  • Componentes React: PascalCase (ej: ProductsList(), AdminLayout())
  • Interfaces de TypeScript: PascalCase (ej: ApiProduct)
