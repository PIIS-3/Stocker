🏛️ Arquitectura Global del Sistema

Por qué hemos separado la aplicación en micro-capas y cómo fluyen los datos desde la Pantalla hasta el Disco Duro.

El Flujo de Vida de un Dato (Data Flow)

Cuando un usuario guarda un producto en el navegador, la petición atraviesa 5 capas estrictas. Esta separación nos permite aislar responsabilidades (Separation of Concerns), facilitando el testing y evitando el "Código Espagueti".

1. Capa de Interfaz (Vistas React) Frontend | JSX

Dónde: frontend/src/pages/ y components/

Las vistas no saben cómo se guardan los datos. Solo recolectan el estado de los inputs (ej. con `useState`) y renderizan el diseño Tailwind. Delegan la responsabilidad llamando a un Servicio estático al hacer submit.

2. Capa de Servicios Locales Frontend | Axios

Dónde: frontend/src/services/

Módulos como products.service.ts encapsulan las llamadas Axios api.post(). Son el único puente que conoce la IP o la URL del Backend. Gracias a la instancia central en api.ts, inyectan automáticamente el Token de Autorización.

3. Enrutadores y Endpoints HTTP Backend | FastAPI

Dónde: backend/app/api/endpoints/ y router.py

Son los "Guardias de Seguridad". Reciben el JSON del frontend, y usando Pydantic (Modelos Create/Response), validan estrictamente que contenga los campos correctos. Si el JSON está mal, lanzan un Error 422 Automático. Si está bien, delegan la lógica a la capa CRUD.

4. Capa Lógica CRUD Backend | Python Puro

Dónde: backend/app/crud/

Funciones aisladas que contienen la "Lógica de Negocio". Transforman los datos, calculan totales, gestionan stock y deciden cómo interactuar con los Modelos SQL. Nunca saben que son parte de una API, no manejan Requests HTTP, solo puras variables Python. Son la capa más fácil de testear (Pruebas Unitarias).

5. Capa de Modelos (ORM) Backend | SQLModel / Postgres

Dónde: backend/app/models/ y database.py

Traducen las Clases Python que recibió el CRUD en verdaderas sentencias SQL instanciando la Sesión segura y forzando los Commits en el motor PostgreSQL remoto.
