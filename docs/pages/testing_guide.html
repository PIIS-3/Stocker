🧪 Guía de Testing y QA Automático

Describe las herramientas de calidad del proyecto: tests automáticos, checks de formato y cómo ejecutarlos manualmente.

1. GitHub Actions — QA Automático en Pull Requests

Los siguientes workflows se ejecutan automáticamente en cada PR y bloquean el merge si fallan.

🔧 ci-backend.yml Activo

Verifica la integridad del backend Python en dos fases que deben pasar en orden:

  1. Tests (pytest tests/): Ejecuta toda la suite de tests del directorio backend/tests/. Si hay algún test roto, el workflow falla aquí y la fase de Docker no se ejecuta (needs: test).
  2. Build Docker: Construye la imagen Docker del backend para verificar que el Dockerfile es correcto y que el contenedor arranca sin errores. No sube la imagen al registro (solo valida).

Cómo ejecutar los tests manualmente:

cd backend
export PYTHONPATH=$PYTHONPATH:$(pwd)
pytest tests

⚛️ ci-frontend.yml Activo

Verifica que el frontend React/TypeScript compila sin errores:

  1. Instala Node.js y las dependencias con caché de GitHub Actions.
  2. Ejecuta npm run build: primero comprueba los tipos con TypeScript (tsc -b) y luego compila con Vite. Si hay algún error de tipo o sintaxis, el build falla.

Cómo ejecutar el build manualmente:

cd frontend
npm run build

📐 lint-frontend.yml Activo

Verifica que el código del frontend sigue los estándares de formato y estilo:

  1. ESLint (npm run lint): Detecta errores de lógica, variables sin usar y malas prácticas de React/TypeScript. Configurado en eslint.config.js.
  2. Prettier (npm run format:check): Verifica que el formato visual del código (comillas, ancho de línea, etc.) cumple las reglas de .prettierrc. Solo comprueba, no modifica archivos.

Cómo ejecutar manualmente:

cd frontend
npm run lint              # Ver errores de lógica
npm run lint -- --fix     # Corregir automáticamente lo posible
npx prettier --write .    # Aplicar formato automáticamente
npm run format:check      # Verificar que el formato es correcto

🌍 lint-global.yml Activo

Verifica que todos los archivos del repositorio (no solo el frontend) cumplen las reglas básicas de .editorconfig: indentación correcta, finales de línea LF, línea vacía final.

No requiere ningún comando manual extra; tu editor lo aplica automáticamente si tienes instalada la extensión EditorConfig.

2. Tests Backend (Pytest)

Los tests del backend están en backend/tests/ y usan pytest con el cliente de tests de FastAPI.

Estructura de tests

backend/tests/
├── conftest.py       # Fixtures compartidos: sesión de BD de test, cliente HTTP, etc.
└── test_stores.py    # Tests de los endpoints de tiendas (GET, POST, PATCH, DELETE)

Comandos disponibles

ComandoDescripción
pytest tests/ Ejecuta todos los tests.
pytest tests/ -v Modo verbose: muestra el nombre de cada test y si pasa o falla.
pytest tests/test_stores.py Solo ejecuta los tests de tiendas.
pytest tests/ -k "test_create" Solo ejecuta los tests cuyo nombre contenga "test_create".
pytest tests/ --tb=short Muestra un traceback corto cuando un test falla (más legible).

Qué se testea actualmente

  • GET /stores/ → Devuelve lista de tiendas (200).
  • POST /stores/ → Crea una tienda y verifica que los timestamps se generan (201).
  • POST /stores/ duplicado → Devuelve 409 si el nombre ya existe.
  • GET /stores/{id} → Devuelve una tienda por ID (200) o 404 si no existe.
  • PATCH /stores/{id} → Actualiza parcialmente y verifica que updated_at cambia.
  • DELETE /stores/{id} → Elimina y devuelve el registro (200) o 404 si no existe.

3. Tests Frontend Planificado

Vite incluye soporte nativo para Vitest, que es mucho más rápido que Jest y no requiere configuración extra.

Herramientas previstas

  • Vitest: Runner de tests nativo de Vite. Misma API que Jest.
  • React Testing Library: Para testear componentes desde la perspectiva del usuario (no de la implementación interna).
  • MSW (Mock Service Worker): Para simular respuestas del backend sin necesitar el servidor real.

Qué se planea testear

  • Que el componente Navbar renderiza correctamente con y sin usuario autenticado.
  • Que la pantalla de login muestra errores cuando las credenciales son incorrectas (respuesta 401 mockeada).
  • Que la lista de tiendas muestra un mensaje vacío si el backend devuelve un array vacío.
  • Que los formularios (create/edit) validan los campos antes de enviar la petición.
